# 编译原理：从零手写一门语言的微观结构

如果你想彻底弄碎编程语言的“魔法”，手搓一个迷你编译器（或解释器）是最高效、最具破坏性的学习方式。
这也是大名鼎鼎的《Crafting Interpreters》或《龙书》的核心大纲。

我们将在这个子目录下，用最原汁原味的 C 语言，从零开始（不借助 Lex/Yacc 等生成工具），手写一门名为 **"MiniLang"** 的微型语言。

## 编译器的绝对结构（四大核心组件）

一段源代码（如 `let x = 10 + 2 * 3;`），在计算机看来只是一串毫无意义的 ASCII 字符。要让它执行，我们必须徒手构建以下四级“安检门”：

### 第一关：词法分析器 (Lexer / Scanner)
* **本质**：高级的“分词器”。
* **工作流**：读取纯文本字符串 `L, e, t,  , x,  , = ...`，剔除空格和注释，把有意义的词组打包成 **Token（词法记号）**。
* **输出结果**：`[KEYWORD("let"), IDENTIFIER("x"), ASSIGN("="), NUMBER("10"), PLUS("+"), NUMBER("2"), MUL("*"), NUMBER("3"), SEMI(";")]`
* **难点**：状态机（DFA），处理连在一起的数字如 `123`，处理双字符操作符如 `==` 或 `>=`。

### 第二关：语法分析器 (Parser)
* **本质**：文法结构的缔造者。
* **工作流**：拿到上一步线性的 Token 数组，根据我们预先定好的“语法规则（Grammar）”，把它们拼装成一棵高维度的 **抽象语法树（AST, Abstract Syntax Tree）**。
* **输出结果**：
  ```text
       (=)
      /   \
    (x)   (+)
         /   \
      (10)   (*)
            /   \
          (2)   (3)
  ```
* **难点**：**运算符优先级**，为什么先算乘后算加？因为 Parser 会把 `*` 节点推到 AST 更深（更早执行）的层级。这就是经典的“递归下降解析算法 (Recursive Descent Parsing)”。

### 第三关：语义分析器 (Semantic Analyzer)
* **本质**：逻辑和类型警察。
* **工作流**：拿着 AST 检查逻辑是否合理。变量 `x` 是否已经声明过？字符串能不能和数字相加（类型检查）？
* **输出结果**：一棵经过严格“查户口”的、带有所有类型信息的 AST。

### 第四关分叉点：解释执行 (Interpreter) 或 代码生成 (Code Generator)
* **解释器 (Interpreter)**：直接写一段 C 语言代码，利用递归，当场去遍历计算那棵 AST 的结果。（Python / JS 早期就是这么干的）。
* **编译器 (Compiler)**：把 AST 彻底打碎，翻译成底层的机器汇编指令或者 VM 字节码 (Bytecode)，存成二进制文件。（C / C++ / Java / C# 的做法）。

## 我们的路线图规划
我们先不搞复杂的编译器，我们先做**第一种（Tree-Walk Interpreter 树遍历解释器）**。我们将手搓以下三个文件：
1. `lexer.c`: 搞定 Token 提取
2. `parser.c`: 搞定算术表达式的优先级的 AST 树构建
3. `evaluator.c`: 运行这棵树！
