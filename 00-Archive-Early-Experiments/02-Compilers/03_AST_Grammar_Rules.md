# 抽象语法树 (AST) 的绝对规则：上下文无关文法

AST（抽象语法树）绝对不是一棵随便乱长的野树。
编译器在把线性的 Token 组装成树的时候，手里必须紧紧攥着一本“造树图纸”。在计算机科学里，这本图纸的专业术语叫做 **上下文无关文法 (Context-Free Grammar, CFG)**。

而所有的程序员和语言设计者，都是用一种非常古老的公式写法来定义这套规则的，它叫 **BNF 范式 (Backus-Naur Form)**。

## 为什么需要规则？（解决优先级的世纪难题）
如果你有一串 Token：`2 + 3 * 4`。
如果不设规则，可能会建出两棵完全不同的树，算出来的结果一个是 `14`，一个是 `20`。
所以，规则（文法）存在的最大意义就是：**强制规定谁先结合、谁后结合（优先级），以及从左往右算还是从右往左算（结合性）。**

---

## 我们迷你语言的 AST “三大造树法则”

如果你想让语言懂得先乘除后加减、且懂得括号最高级，你的造树规则必须像“俄罗斯套娃”一样，设计出**严格的等级森严的层级图谱**。

在编译原理中，我们通常把算术表达式拆解成三个等级：**表达式 (Expression) -> 项 (Term) -> 因子 (Factor)**。等级越高的，越在树的顶层（最后算）；等级越低的（如 Factor），越在树的底层（最先算）。

下面就是我们脑海里这门语言的 **BNF 文法规则**（非常像高中数学里的推导规则）：

### 等级 1：表达式 (Expression) —— 最顶级的规则，统管加减法
```text
Expression（表达式） ➡️ Term（项） 后面可能跟着无数个 ( '+' 或 '-' 另一个 Term ) 
```
* **人话翻译**：一段算术表达式，本质上就是由几个大的“项（Term）”通过加减号连起来的。比如 `A + B - C`。因为加减法优先级最低，所以它们要被放在树的最顶层最后算。

### 等级 2：项 (Term) —— 中间等级，统管乘除法
```text
Term（项） ➡️ Factor（因子） 后面可能跟着无数个 ( '*' 或 '/' 另一个 Factor )
```
* **人话翻译**：那上面的 `A`（Term）到底是个什么东西？大写 `A` 其实里面可能包着乘除法。比如 `2 * 3` 整体被视为一个 Term。因为乘除法优先级高，所以它被推向了树的更深一层。

### 等级 3：因子 (Factor) —— 底层不可分割的反叛者
```text
Factor（因子） ➡️ NUMBER（纯数字） 
                  或者
                  '(' Expression ')' （带括号的表达式）
```
* **人话翻译**：什么是最小不可分割的颗粒（因子）？要么是一个干干净净的数字 `42`。要么，是**被括号包裹起来的另一个顶级表达式！**
* **魔法在此诞生**：注意看这里！`Factor` 里面竟然又可以包一个 `Expression`！这就叫**递归文法**。正因为有了这个递归，你的语言才能解析出像 `( 1 + ( 2 * (3 - 1) ) )` 这样无限套娃的疯狂数学表达式。

## 建立 AST 时的执行逻辑（也就是 Parser 的逻辑）

当 Parser (语法分析器) 拿到 `2 + 3 * 4` 时，它是这么比对规则建树的：

1. **查规则 1**：看到 `+` 号，知道两边分别是两个 Term。左边是个 Term（包裹着 `2`），右边是个 Term（包裹着 `3*4`）。此时树根诞生了，就是一个 `+` 节点。
2. **查规则 2**：追究右边的 Term，发现里面有个 `*` 号！所以这个 Term 被分叉成两个 Factor：`3` 和 `4`。
3. **查规则 3**：`2`、`3`、`4` 都已经是纯数字 Factor，不可分割了，变成了树叶。

至此，树叶 `3` 和 `4` 扎得最深，所以执行时一定会最优先被乘起来，然后再和比较浅的 `2` 加起来。
**“优先级”这个虚无缥缈的数学概念，就这样被物理降维成了一棵树的“深度层级”！**
