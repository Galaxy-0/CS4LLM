# 现代 CS 科班本科生能力基准模型 (Capability Matrix)

要评估或锚定一条学习路线是否“达标”，我们必须弄清楚当前互联网大厂（如 BAT、TMD 乃至硅谷 Flag）或者顶尖高校（清北复交/常青藤）对一个合格 CS 毕业生的期望究竟是什么。

目前的 CS 本科培养已经严重两极分化。如果你只是想“期末考试及格及拿到双证”，那门槛极低；但如果你要达到**“具备大厂系统研发潜力的高级毕业生”**水平，标准是非常苛刻的。

我们将这种能力划分为四个维度（也是面试中考察的绝对重点）：

---

## 1. 编程与工程直觉（Coding & Engineering）
这不是指你会写多少种语言的 `Hello World`，而是你对代码的掌控力。
* **语言深度**：精通至少一门高级语言（如 Java, C++, Go），不仅懂语法，还要懂其背后的运行时机制（比如知道 Java GC 怎么回收、C++ 的智能指针原理、Go 的 Goroutine 调度模型）。
* **工程素养**：代码不再是面条式（Spaghetti code）。接手几万行的开源项目，能用纯英文阅读官方文档（Documentation），熟练运用 Git 追踪回滚，会写高质量的单元测试，并且懂得在 Linux 终端下只用键盘开发和调试代码（摆脱对笨重可视化 IDE 的绝对依赖）。

## 2. 也是最大的分水岭：四大底层内功的贯通（Fundamentals）
这也是我们这个 `CS4LLM` 仓库正在攻克的内容。大厂只看你内功深不深，不看你用了哪个流行的花哨框架。
* **数据结构与算法**：手撕代码是基本盘。必须能在白板（不靠编译器纠错）上，20 分钟内默写并调通中等难度（LeetCode Medium）难度的题。比如熟练写出动态规划状态转移方程、图的 DFS/BFS 遍历、手画红黑树/B+树的分裂过程。
* **操作系统能力**：这是决定你能否排查线上诡异 Bug 的关键。你要能立刻反应出：为什么一个 Java 线程死锁了 CPU 利用率飙到 100%？为什么高并发读写文件时响应极慢（缺页中断导致大量磁盘 I/O）？必须深刻理解并发（Concurrency）、锁（Mutex）、虚拟内存分页、进程/线程调度。
* **网络穿透力**：写出 CRUD 接口那是培训班水平。科班生必须回答：TCP 为什么要三次握手和四次挥手？如果服务端突然断电，客户端会发生什么？懂得拥塞控制机制、HTTP/2 多路复用、HTTPS 证书与对称加密协商全流程。
* **数据库底座**：绝不仅限于写 SQL 连表查询。而是要懂：因为磁盘是按页（Page）读取的，如果不用 B+ 树做索引会变多慢？不同的并发事务（脏读、幻读）在底层是如何靠 MVCC（多版本并发控制）和间隙锁（Gap Lock）解决的？

## 3. 系统级造轮子的实战经历（Systems Building）
最顶尖的本科生，他们的简历上不会仅仅写着“用 Spring Boot 做了一个商城（这种叫调包侠）”；而是**“我亲手从零写过一个底层的轮子”**。
* **比如**：你亲手在 C 语言上写过一个内存分配器（`malloc` / `free`）。
* **比如**：像我们刚才那样，不拿现成的高级框架（Nginx/Tomcat），直接用操作系统 Socket API 手撕过一个支持几千人并发的高性能 HTTP Web Server。
* **比如**：完成过像 MIT 6.828/6.S081 这种把一个微型操作系统（xv6）补充到能跑起来的疯狂大作业。

## 4. 前沿架构视野（Architecture Vision）
虽然本科不需要精通架构，但必须具备对**分布式系统（Distributed Systems）**的常识。
* 知道如果一台机器存不下 100 亿条数据该怎么办（Sharding / Consistent Hashing）。
* 知道当三台主从数据库因为网线被挖断（网络分区）时，为什么没办法既保证它们数据一模一样，又立刻响应客户端（大名鼎鼎的 CAP 定理），以及选举算法（Raft / Paxos）的运行机制。

---

> **总结：**
> 当前 CS 科班的最高水平，**并不是什么都学、什么框架都会调**。
> 而是：**拿着一把最素的“生锈铁剑”（如 C / C++），直接站在冰冷的“裸奔机器”（Linux 内核层面）上，用自己徒手写出的精妙算法（双指针、树遍历），去迎战海量的请求和高并发危机。**
> 这，也是咱们前面这半个多小时在 `CS4LLM` 仓库里所体验的核爆级密度的学习路径的最终目的！
